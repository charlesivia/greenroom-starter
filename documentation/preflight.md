# Pre-flight Deal Review Handoff

## 1. Overview

The implemented slice is **Pre-flight Deal Review**.

The product thesis is that settlement disputes often come from ambiguous deal prose before show night, not just bad 2am math. The prototype creates a place to detect narrow ambiguity classes, send a clarification to the agent, record the reply, and carry the resolved interpretation into settlement review.

## 2. What Shipped

- `deal_clarifications` schema: one new table for detected deal ambiguities, suggested clarification text, status, agent reply, resolution metadata, and citations.
- `marketing_recoup_cap` detector: detects deal prose where a marketing recoup intersects with an expense cap.
- `bonus_structure_drift` detector: detects bonus-related deal prose when `bonuses_json` is missing.
- Show detail Pre-flight deal review section: server-rendered section after Deal terms and before operational sections.
- Send to agent action: transitions a pending clarification to `sent_to_agent`.
- Record reply action: captures `agentReplyText`, marks the clarification `resolved`, sets `resolvedVia = in_app_reply`, and revalidates the show page.
- Settlement handoff block: shows resolved pre-flight interpretations on `/shows/[id]/settle` before recoups/signoff context.
- Demo and verification scripts for data checks and detector regression.

## 3. Detector Behavior

### `marketing_recoup_cap`

Detects:
- Deal prose that references marketing or recoup language.
- A structured `expenseCap` on the deal.
- Ambiguity around whether the marketing recoup sits inside or outside the expense cap.
- Relevant past marketing recoup disputes at the same venue, excluding the target show itself.

Does not detect:
- Generic expense caps with no marketing or recoup language.
- Production overages.
- Hospitality caps.
- Agent risk patterns.
- Whether the settlement math should change.

Severity:
- `high` when there is complete past dispute context with agency, date, and disputed amount.
- `medium` fallback only when precedent context is missing or incomplete.

### `bonus_structure_drift`

Detects:
- `bonusesJson` is null.
- Deal prose contains bonus-related language such as `bonus`, `bonuses`, `walkout`, `threshold`, `gross above`, `over $`, or `pot`.

Does not detect:
- Deals where structured `bonusesJson` already exists.
- Actual bonus math.
- Whether the bonus should stack with a guarantee or percentage payout.
- Past dispute precedent.
- Agent risk patterns.

Severity:
- Always `medium`, because the risk is missing structured data for settlement, not a demonstrated venue-level dispute pattern.

### Deterministic Gates

Both detectors use deterministic gates around the LLM call.

Pre-LLM gates:
- `marketing_recoup_cap` skips the model unless deal prose contains `recoup` or `marketing` and `expenseCap` exists.
- `bonus_structure_drift` skips the model unless `bonusesJson` is null and the prose contains a bonus-related signal.

Post-LLM gates:
- `detectedPhraseFromDeal` must be a literal substring of `dealNotesFreetext`.
- `marketing_recoup_cap` phrases must still contain `recoup` or `marketing`.
- `bonus_structure_drift` phrases must still contain a bonus-related term.
- Citations must come from the supplied context only.

Suggested clarification copy is templated in code, not generated by the model.

## 5. Debug Moments

- The first marketing detector pass produced false positives by firing on `expense_cap` alone.
- The fix was a two-layer deterministic gate: require `recoup` or `marketing` before the LLM call, then require the returned phrase to also contain `recoup` or `marketing`.
- The marketing context initially allowed a show to cite itself as precedent. The target show is now excluded from its own `pastDisputes`.
- The bonus detector was initially over-scoped with `pastBonusDisputes` and a broader persistence refactor. That was narrowed back to the actual class: bonus prose exists, but `bonuses_json` is missing.
- Bonus citations are intentionally empty in v1 because the data overlap is thin and the risk is missing structure, not precedent.

## 7. Commands

Reset the local database:

```bash
npm run db:reset
```

Run marketing detector verification:

```bash
npm run preflight:check -- <showId>
```

Run marketing detector and persist detected risks:

```bash
npm run preflight:check -- <showId> --persist
```

Run bonus detector verification:

```bash
npm run preflight:check:bonus -- <showId>
```

Persist the demo bonus clarification for `show_0167` after reset:

```bash
npm run preflight:persist:bonus-demo
```

npx tsc --noEmit

## 9. Known Limitations / V2

- No email integration. Send to agent is a product state transition only.
- No dismiss flow.
- No queue view.
- Settlement math does not consume clarifications yet.
- The settlement handoff is visible provenance, not a calculator change.
- The bonus detector does not parse actual bonus math.
- `agent_risk_pattern` is not built.
- Production overage ambiguity is represented by the schema only indirectly; there is no detector for it.
- All-clear means no recorded clarifications for the deal, not live detection.
- The active bonus demo case requires `npm run preflight:persist:bonus-demo` after `npm run db:reset`.
- `npm run lint` has unrelated existing failures outside the pre-flight implementation path.
